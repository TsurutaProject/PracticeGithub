# Githubのリポジトリを立てた時にまずやること

## gitとGithubを関連づけ、Github上の進捗をローカルに持ってくる

まず、gitとGithubを関連づける必要があります。
その時に同時にGithub(リモート)上の進捗をローカルに持っていくことができます。
<br>
以下のコマンドを使用するディレクトリ上で叩きます。
```
git clone <Githubのurl> .
```
<br>
今回は
<br>

```
git clone https://github.com/TsurutaProject/PracticeGithub.git .
```

<br>

と叩きます。

すると、gitとGithubを関連つけると同時に、Github上の進捗をローカルに持ってくることが可能です。

## branchの作成と移動
まず何も考えずに以下のコマンドを叩いてください。
<br>
```
git branch 
```
<br>
すると
<br>

```
*main
```

<br>
という表示が出るかと思います。
この場合、「今はmainブランチにいる」ということを表しています。
基本的にGithubを使うときの共同開発はmainブランチはいじらないということになっています。
そのため、以下のコマンドを叩き、ブランチを作成し移動する必要性があります。<br>

```
git checkout -b feature-<自分が使用するbranch>
```
<br>
<自分が使用するbranch>の名前はアルファベットを使用し、自分の名前を入れてもらいたいです。<br>

ex) 
```
git checkout -b feature-norimaki
```
<br>
そうするとnorimakiというブランチを作成し、移動することができます。

## 進捗を自分のブランチに記録する(push)
自分のブランチを立てることができたら、<br>
<自分の名前>.html<br>
というファイルを作成していただきたいと思います。<br>
その後いつもの
<br>
```
git add .
```
<br>

```
git commit -m "<コミットメッセージ>"
```

<br>
をしてもらいます。
なお、コミットメッセージについては共同開発では色々と決まりがあります。
その決まりについては下記にあるものを参考にしてください。
commitし終わったら、以下のコマンドを叩きます。
<br>

```
git push origin <自分のブランチ>
```

<br>
このコマンドを叩くと自分のブランチにローカルでの進捗をGithubに上げることができます。
この時
<br>

```
origin <自分のブランチ>
```

<br>
の部分は<br>
# 絶対に
<br>
忘れないようにお願いします。
でないと、mainブランチに進捗が上げられてしまう恐れがあり、大変なことになります。
<br>

```
git push origin <自分のブランチ>
```

<br>
を忘れないようにお願いします。

## 自分のブランチの進捗をmainにマージする(pullリクエスト)
何かしらの機能を実装したら、mainブランチの方にその進捗をマージする必要があります。
その時にpullリクエストというものをオーナーの方に送る必要があります。<br>
pushをした後にGithubで自分のブランチを見るとpullRequestと書いてあるボタンがあると思います。
そのボタンを押すと、titleとやったことを書く画面になります。
そこで、自分のやったことのタイトルと、自分のやったことの概要を書く必要があります。<br>
やったことの概要についてはマークダウン方式で書きますが、マークダウン方式はここで記述する必要はない
はずなので、自分で調べてください。<br>
無事書き終わったら、Create PullRequestのボタンを押してpullリクエストを送ります。<br>
その後、コードレビューなどを通してmainブランチにオーナーがマージをします。

## マージしたmainブランチをローカルにマージする(pull)
基本的に共同開発での進捗はmainブランチが基準となります。
そのため、mainブランチが更新された時にローカルにその進捗を保存する必要があります。
その時に以下のコマンドを入力します。
<br>

```
git pull origin main
```

<br>
このコマンドを入力することで、進捗を最新のものにすることができます。
逆にこのコマンドを叩いていないと、コンフリクトということが起きてしまい、大変なことになってしまう場合があるので
更新されたらその都度pullすることをお勧めします。
<br><br>
これらが、Githubで共同開発をするときにこれだけやっておけばとりあえず大丈夫だという内容です。

# 共同開発をする上で、便利であること

## issuesの存在
あまり、オーナー以外が使うことはないですが、共同開発をする上で問題になっていること(バグなど)や、実装して欲しいことを書くときに用いるものです。
issuesはpullリクエストのように#Noで管理されています。また、担当者についてやラベルなどで管理できます。
issuesが解決する前はopenという状態になり、解決すればcloseの状態になります。
またissues2のように他のissuesやpullリクエストをメンションすることも可能です。

## コミットメッセージ

基本は短縮バァジョンでいいと思います。

### Prefix（コミットメッセージ記法）

| Prefix         | 説明                                                       |
| -------------- | ---------------------------------------------------------- |
| init           | first commit                                               |
| chore          | ビルドプロセスやドキュメント生成などの補助ライブラリの変更 |
| add            | 追加(ファイル、機能)                                       |
| feat, feature  | 追加(機能、ファイル)                                       |
| update         | 機能修正(バグではない)                                     |
| perf           | パフォーマンスを向上させるためのコード変更                 |
| refactor       | 整理(リファクタリング等)                                   |
| change         | 仕様変更                                                   |
| fix            | バグ修正                                                   |
| remove, delete | 削除(ファイル、機能)                                       |
| revert         | 変更取り消し                                               |
| test           | 不足しているテストの追加や既存のテストの修正               |
| docs           | ドキュメンテーションのみの変更                             |
| merge          | ブランチをマージ                                           |
| sleep          | 寝た w                                                     |

### Prefix(短縮 Ver)

| Prefix         | 説明                   |
| -------------- | ---------------------- |
| add            | 追加(ファイル、機能)   |
| remove, delete | 削除(ファイル、機能)   |
| update         | 機能修正(バグではない) |
| fix            | バグ修正               |

#### コミットメッセージの例

- init
- chore: React と依存関係を追加
- add: ユーザー登録機能を追加
- sleep: I'm so sleepy
